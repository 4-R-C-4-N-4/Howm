    ### 11.9 Zone System

    Blocks are subdivided into **zones** — sub-regions that carry spawn mode, density, and object affinity. Zones are the fundamental unit of object placement for all object types.

    #### Zone generation

    Zones are generated by seeded Voronoi subdivision within the block polygon:

    ```
    zone_count = max(2, min(12,
    floor(block.area / CONFIG.ZONE_AREA_BASE)
    + floor(popcount_ratio × CONFIG.ZONE_ENTROPY_BONUS)))
    ```

    Zone seed points:

    ```
    for z in 0..zone_count:
    zone_pt_seed = ha(cell_key ^ block.idx ^ 0x7a3f ^ z)
    zone_seed_pt = point_in_polygon(block.poly, zone_pt_seed)
    ```

    Zone polygons are the Voronoi cells clipped to the block polygon. Each zone carries:

    ```
    zone {
        idx:              uint
        seed:             uint32        // ha(cell_key ^ block.idx ^ zone_idx)
        polygon:          [point]
        centroid:         point
        area:             float
        density:          float         // 0.0–1.0; ha(zone.seed ^ 0x1) / 0xFFFFFFFF
        affinity:         [role_id]     // preferred roles; from zone.seed ^ 0x2
        reseed_interval:  uint64        // ms; controls spawn position stability
    }
    ```

    | Block type | `reseed_interval` | Effect |
    |------------|-------------------|--------|
    | `building` | `∞` (`0xFFFFFFFFFFFF`) | Fixtures never re-seed |
    | `park` | `86_400_000` (24 hours) | Flora scatters daily |
    | `plaza` | `∞` | Fixtures and ornaments fixed |
    | `water` | `∞` | Water structures fixed |
    | `riverbank` | `3_600_000` (1 hour) | Minor flora shifts hourly |

    OK. Now I have the full picture. Let me write up a clear assessment.
    The spec defines this pipeline:

    IP → Cell → Voronoi District → Roads + Rivers → PSLG → Half-Edge Face Extraction → BLOCKS → Zones → Buildings/Fixtures/Flora within zones


    The critical invariant: Buildings exist within blocks. Blocks are the faces BETWEEN roads. Roads are the boundaries that separate blocks. A building should NEVER overlap a road because buildings live inside block polygons, and block polygons are bounded by road segments.

    Here are the actual bugs, in order of severity:

    Bug 1 (ROOT CAUSE): The PSLG half-edge face extraction fails for most districts. I showed earlier that 7 of 10 test IPs produce only 1 block despite having 2-8 road segments. The half-edge algorithm in blocks.rs extract_faces() is not correctly finding closed faces. This means for most districts, the entire district is treated as ONE block, and all buildings are placed via Voronoi subdivision of that single giant block — completely ignoring roads.

    The half-edge bugs:
    - Road terminals sit on boundary edges but may not properly split those edges into sub-segments due to snap tolerance
    - MeetingPoint road junctions float in the interior without connecting back to the boundary, so they can't form closed faces
    - The next-pointer angle logic may have the rotation direction wrong (§9.3 says "counter-clockwise around B from reverse direction B→A" but the code comments say "CW rotation")

    Bug 2 (SYMPTOM): Building plots clip to bounding box, not actual polygon. This was the bug I found — clip_polygon uses the bbox. My fix projects outside vertices to the nearest boundary edge, which constrains them but doesn't properly intersect. The real fix is moot once Bug 1 is fixed because blocks will be smaller proper polygons where the bbox approximation is much closer to the actual shape.

    Bug 3 (DESIGN): Zones are not being used as the placement unit. The spec says zones (§11.9) are "the fundamental unit of object placement for ALL object types." Fixtures, flora, and creatures should spawn within zones. Buildings should go in building-typed blocks. But the current code has generate_buildings doing its own Voronoi subdivision of the block directly (§12.3), separate from the zone system. This is actually correct per spec — zones are for fixtures/flora/creatures, while buildings have their own plot subdivision pipeline (§12.1). But it means buildings don't respect zone boundaries, only block boundaries.

    The fundamental fix needed: Rewrite the half-edge face extraction in blocks.rs so that roads actually subdivide the district into multiple blocks. Once blocks are correct, buildings will naturally be constrained to the spaces between roads, because each block polygon IS the space between roads. My clipping fix and the map boundary visibility fix are band-aids — the real issue is that the PSLG isn't producing faces.
    Two bugs found and fixed in blocks.rs:

    1. Boundary edges weren't being split at road terminal points. Road terminals sit ON boundary edges, but the boundary was stored as its original vertex chain without inserting the road connection points. The half-edge graph had road endpoints that weren't topologically connected to the boundary chain, so no closed faces could form. Fixed by walking each boundary edge and splitting it at every PSLG vertex that lies on it.

    2. The signed area filter was inverted. Interior block faces have positive signed area in our coordinate system, but the code was discarding positive and keeping negative (the exterior face). Flipped the filter.

    The maps now show roads clearly dividing districts into multiple blocks, with buildings properly contained within those blocks. No buildings cross roads.
